## Terminology & Word Choice

### Technical jargon

Use simple, user-friendly language whenever possible. Avoid technical terms unless they are necessary for accuracy.

**Preferred:**

* Sign in
* Select
* Account
* App
* Support service

**Avoid:**

* Authenticate
* Execute
* Application
* User credential repository
* Intervention platform

> **Note:** Use **Select** as the standard verb for interacting with buttons, menus, links, and other interface elements. Reserve **Open** for launching apps or features, **Enter** for typing information, and **Swipe** for touch gestures.

### Preferred terminology

| Preferred Term        | Discouraged Term                                          |
| --------------------- | --------------------------------------------------------- |
| Child                 | Kid                                                       |
| User                  | Client                                                    |
| Support service       | Intervention                                              |
| Sign in               | Log in to                                                 |
| Select                | Click / Tap*                                              |

*Use **Tap** only when platform-specific wording is required. Otherwise, prefer **Select** throughout the documentation for consistency.

## Active vs. Passive Voice

Use **active voice** whenever possible.

**Preferred:**

* Open **My Diary**.
* Select **Helping Hand**.
* Scroll through the article.

**Avoid:**

* **My Diary** can be accessed through the menu.
* The article can be viewed by selecting the title.

---

## Document Structure

Use the following structure for user guides whenever applicable:

1. **Title** (action-oriented)
2. **Overview**
3. **Before You Begin** (if prerequisites exist)
4. **Opening the Feature**
5. **Using the Feature** (numbered procedures)
6. **Troubleshooting** (optional)
7. **Additional Notes** (optional)

Avoid separate **Purpose** and **Expected Result** sections in end-user documentation unless required by the project.

---

## Writing Procedures

* Begin each step with an action verb.
* Write one action per numbered step whenever possible.
* Use numbered lists for sequential actions.
* Use bullet lists only for options, examples, categories, or tips.
* Include a brief outcome statement after a procedure when helpful.

**Example:**

1. Open the **MyChild Helpline** app.
2. Select **Information Kiosk**.
3. Select **My Rights**.

The app displays the selected page.

## Punctuation & Mechanics

### Lists

Rules:

* Start each bullet point with a capital letter.
* Use parallel sentence structure.
* End complete-sentence bullet points with a period.
* Do not use periods for short fragments.

### Oxford comma

Use the Oxford comma consistently.

**Correct:**

* The app includes **Helping Hand**, **My Diary**, and **Call Us**.

**Incorrect:**

* The app includes **Helping Hand**, **My Diary** and **Call Us**.

---

## Formatting UI Elements

* Use **bold** for buttons, menu items, tabs, screen names, icons, and feature names.
* Use sentence case for headings.
* Use block quotes for notes and warnings.

**Example:**

> **Note:** An internet connection is required to access online resources.

> **Important:** Deleted diary entries cannot be recovered.

## Style Guide Checklist

Before publishing documentation, verify that:

* Active voice is used whenever possible.
* Procedures follow the standard document structure.
* Headings use sentence case.
* Numbered lists are used for sequential actions.
* Mental health-related content uses respectful and supportive language.
* Acronyms are defined on first mention.
* Bold text is used only for UI elements, feature names, and important notices.
* Notes and warnings use blockquote formatting.
* Lists follow capitalization and punctuation rules.
* The Oxford comma is used consistently.
* Approved terminology is used throughout the document.
* **Select** is used consistently for user interface interactions unless a platform-specific action requires another verb.
